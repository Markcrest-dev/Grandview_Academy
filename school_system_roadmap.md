## ⚠️ Design Philosophy — CRITICAL: Read Before Writing Any UI Code

This website must look like it was built by a **skilled human developer** — not generated by AI.
The following rules are non-negotiable and apply to every single page and component.

### ❌ What to NEVER Do
- **No color gradients** — not on buttons, backgrounds, cards, or anywhere else
- **No repetitive color blocks** — avoid slapping the primary color on every section back to back
- **No excessive icons or emoji** in the UI — one or two where genuinely needed, never decorative
- **No stock-photo collage hero sections** with overlaid gradient text
- **No rounded pill buttons on everything** — use consistent, intentional button styles
- **No card grid overload** — not every section needs to be a 3-column icon card grid
- **No AI-typical section patterns** — e.g. hero → 3 feature cards → testimonial → CTA → footer, repeated on every page
- **No excessive shadow and glow effects** on components
- **No animations on everything** — motion should be intentional and subtle, not on every element
- **No generic "Lorem Ipsum" placeholder layouts** shipped as final design

### ✅ What to Always Do
- **Use whitespace deliberately** — give elements room to breathe
- **Keep typography consistent** — two fonts maximum, used with clear hierarchy
- **Let content drive layout** — the layout should serve the content, not the other way around
- **Use the color palette sparingly** — navy and gold are accents, white and grey carry the page
- **Write real copy** — placeholder text in UI components should reflect actual Grandview Academy content
- **Be intentional with every component** — if you cannot explain why it is there, remove it
- **Make it feel institutional** — clean, serious, and trustworthy like a real school website
- **Design mobile first** — build for small screens first, then scale up

## 1. Project Overview

**What we are building:**
A complete, multi-level School Management System consisting of:
- A **public-facing school website** (visible to anyone)
- **Five secure role-based portals** (accessible only to authorised users)

**Who it serves:**
| Audience | Purpose |
|---|---|
| General public / prospective families | Learn about the school |
| Admin | Manage the entire system |
| Teaching Staff | Manage academics and students |
| Non-Teaching Staff | Manage their department operations |
| Parents / Guardians | Monitor their child |
| Students | Access their academic information |

## 2. Institution Structure

This system is designed for an institution that operates **three levels under one organisation:**

- 🟡 **Primary School**
- 🔵 **Secondary School**
- 🟢 **University**

### How the website handles multiple levels:
We use **Option A — One Website, Separate Sections.**

All three levels live under one domain with their own dedicated sections:

```
yourschool.edu.ng/primary
yourschool.edu.ng/secondary
yourschool.edu.ng/university
```

Each section has its own:
- Academic programmes
- News and events
- Staff directory

But they all share:
- The same homepage and branding
- The same portal login system
- The same admin backend

## 3. System Architecture

The system is built in three layers:

```
┌─────────────────────────────────────┐
│           FRONTEND (React)          │  ← What users see
│  Public Website + 5 Portal UIs      │
└────────────────┬────────────────────┘
                 │  API Calls
┌────────────────▼────────────────────┐
│        BACKEND (Node + Express)     │  ← The brain
│  Auth, Roles, Business Logic, API   │
└────────────────┬────────────────────┘
                 │  Queries
┌────────────────▼────────────────────┐
│        DATABASE (Supabase/          │  ← Where data lives
│         PostgreSQL)                 │
└─────────────────────────────────────┘
```

This is called a **Decoupled Architecture** — the frontend and backend are two separate projects that communicate through an API. This is clean, scalable, and professional.

## 5. Portal System — All 5 Portals

> **Golden Rule:** Roles are assigned by the system/admin — NEVER chosen by the user themselves.

---

### 🛡️ Portal 1 — Admin Portal

**Access Level:** Super User — full system control
**Who uses it:** School IT officer, Director, or designated system administrator

**Capabilities:**
- Create, edit, deactivate accounts for ALL users
- Assign and change roles (student, staff, parent, etc.)
- Set permissions — what each role can and cannot access
- View reports and analytics across the entire institution
- Manage academic years, terms, classes, departments
- Access all other portals in read-only/audit mode
- View full audit logs of all system activity
- Manage school-wide announcements and notices

**Sub-Admin Support:**
The admin can create **Sub-Admins** with limited powers, for example:
- Registrar Sub-Admin — manages student enrolment only
- Finance Sub-Admin — manages fees only

---

### 👨‍🏫 Portal 2 — Teaching Staff Portal

**Access Level:** Academic operations
**Who uses it:** All classroom teachers across primary, secondary, and university levels

**Capabilities:**
- View assigned classes and student lists
- Take and manage attendance
- Upload and manage grades / results
- Upload lesson notes and learning materials
- View personal timetable and schedule
- Communicate with students and parents
- Submit end-of-term reports

---

### 🗂️ Portal 3 — Non-Teaching Staff Portal

**Access Level:** Department-specific operations
**Who uses it:** Bursar, Librarian, School Nurse, Security, Admin Officers

> Non-teaching staff only see what is relevant to their own department.

| Staff Type | What They Access |
|---|---|
| **Bursar / Accountant** | Fee payments, invoices, financial records |
| **Librarian** | Book inventory, borrowing and return records |
| **School Nurse** | Student health and medical records |
| **Admin Officer** | General records, document management |

---

### 👪 Portal 4 — Parent / Guardian Portal

**Access Level:** Read-only view of their child's data
**Who uses it:** Parents and guardians of enrolled students

**Capabilities:**
- View child's attendance record
- See grades and download result sheets
- Pay school fees online
- Read school announcements and notices
- Message teachers directly
- View child's timetable
- Receive notifications (e.g. absenteeism alerts)

**Multi-Child Support:**
If a parent has more than one child enrolled, they see all children under one login and can switch between their profiles.

---

### 🎓 Portal 5 — Student Portal

**Access Level:** Personal academic data only
**Who uses it:** All enrolled students (primary, secondary, university)

**Capabilities:**
- View personal timetable
- Check own attendance record
- View grades and download result sheets
- Access learning materials uploaded by teachers
- Read school announcements
- Check fee payment status
- *(University level only)* Course registration, exam card generation

> Students can ONLY see their own data — never another student's.

## 6. Security Design

### How Role Security Works

Accounts are **never self-created by users.** The school admin creates every account and assigns its role. Users only receive login credentials.

**Role-Based Access Control (RBAC):**
Every account has a role stored at the database level. The server checks this role on every request — it is not just hidden buttons on the screen, it is enforced at the API level.

---

### Admin Portal — Extra Security Layers

| Security Measure | How It Works |
|---|---|
| **Hidden Login URL** | Admin login page is not linked publicly. You must know the URL to find it |
| **Two-Factor Authentication (2FA)** | After password, a one-time code is sent to admin's phone/email |
| **Strong Password Policy** | Long, complex passwords required — enforced by the system |
| **IP Restriction (optional)** | Login only allowed from school network or approved devices |
| **Session Timeout** | Auto logout after 5 minutes of inactivity |
| **Login Attempt Limit** | Account locks after 3–5 failed attempts. Real admin receives an alert |
| **Full Audit Logs** | Every admin action is recorded — what changed, when, and by whom |

### Password Security
- All passwords are **encrypted with bcrypt** before being stored
- Plain text passwords are NEVER stored in the database
- Even the admin cannot see a user's password — only reset it

### Data Security
- Sensitive data (grades, health records, fees) is **encrypted**
- All data transfer happens over **HTTPS** (encrypted connection)
- Students cannot see other students' data under any circumstance

---

### The Unique ID Verification System

Every account is tied to a school-issued ID that only the school controls:

| Role | Identifier |
|---|---|
| Student | Admission Number |
| Teaching Staff | Staff ID |
| Non-Teaching Staff | Staff ID |
| Parent | Linked to their child's Admission Number |
| Admin | Personally issued by the institution director |

This means even if someone tries to create a fake account claiming a different role, they would not have the valid ID to complete it.

## 7. Tech Stack

### Final Chosen Stack

| Layer | Technology | Reason |
|---|---|---|
| **Frontend** | React.js + Vite | Lightweight, fast on low-spec computers, industry standard |
| **Styling** | Tailwind CSS | Speeds up design, clean and responsive |
| **Backend** | Node.js + Express.js | Handles API, authentication, and business logic |
| **Database** | PostgreSQL via Supabase | Structured relational data, perfect for school systems |
| **Authentication** | Supabase Auth + JWT | Handles roles, sessions, and security efficiently |
| **File Storage** | Cloudinary | Stores uploaded images, documents, and result sheets |
| **Frontend Hosting** | Vercel | Free tier, fast, easy deployment |
| **Backend Hosting** | Render or Railway | Free tier available, reliable |

> **Why React over Next.js:** React with Vite is significantly lighter on system resources. Next.js (which is built on React) requires more RAM and has slower startup times. React alone is the correct learning foundation and produces the same quality output for this project.

---

### How the Stack Connects

```
[ React + Vite ]  ──── HTTP/API calls ────►  [ Express.js ]
  (Frontend)                                   (Backend)
                                                   │
                                                   ▼
                                           [ Supabase DB ]
                                           (PostgreSQL)

Uploaded files ──► [ Cloudinary ]
Deployed on  ──► [ Vercel (frontend) + Render (backend) ]
```

## 8. Database Design Plan

> Full schema to be designed in detail before coding begins. Below is the high-level relationship map.

### Core Entities (Tables)

- **Users** — all accounts (role field determines type)
- **Students** — student-specific profile data
- **Staff** — staff-specific profile data
- **Parents** — linked to one or more students
- **Classes** — each class belongs to a level (primary/secondary/university)
- **Subjects** — each subject linked to a class and a teacher
- **Attendance** — per student, per day, per class
- **Grades / Results** — per student, per subject, per term
- **Fees** — per student, payment status, amount, due dates
- **Announcements** — school-wide or level-specific notices
- **Timetable** — per class, per day
- **Audit Logs** — every admin action recorded

### Key Relationships

```
Student ──────► Class ──────► Subject ──────► Teacher (Staff)
   │
   └──────► Parent (linked via admission number)
   └──────► Attendance records
   └──────► Grade records
   └──────► Fee records
```

> Rule: Use **PostgreSQL** (relational database) because school data has clear structure and relationships. MongoDB is NOT suitable here.

## 13. Rules to Avoid Errors

These are the non-negotiable rules to follow throughout development to keep the project clean:

### Planning Rules
- ✅ Never write code before the database schema is finalised
- ✅ Never build a portal before the authentication system is done
- ✅ Always wireframe a page before building it

### Security Rules
- ✅ Never allow users to assign their own role — always admin-assigned
- ✅ Always enforce role checks on the **backend**, not just the frontend
- ✅ Never store plain text passwords — always use bcrypt
- ✅ Always use HTTPS — never HTTP in production
- ✅ Keep the admin login URL hidden from public pages
- ✅ Always validate and sanitise user inputs (never trust what a user types)

### Code Rules
- ✅ Use `.env` files for all secrets — never hardcode API keys or passwords
- ✅ Never commit `.env` files to GitHub — add them to `.gitignore`
- ✅ Keep frontend and backend as separate projects in separate folders
- ✅ Use meaningful variable and function names
- ✅ Comment your code — future you will thank present you
- ✅ Test each feature fully before moving to the next phase

### Database Rules
- ✅ Design the full schema before creating any tables
- ✅ Use foreign keys to properly link related tables
- ✅ Never delete data permanently — use a soft delete (mark as inactive)
- ✅ Back up the database regularly once live
